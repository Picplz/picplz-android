# ui-ask-question-interview

UI 디테일 조정을 **질문 UI 기반 의사결정 라운드**로 잠그는 스킬.

핵심 목적은 세 가지다.

1. 애매한 디자인 피드백을 구현 가능한 수치/구조로 바꾼다.
2. 첫 라운드에서 놓친 구조적 의도를, 후속 라운드에서 다시 캐낸다.
3. 질문이 많아져도 사용자가 피로하지 않도록 **라운드별로 압축**한다.

---

## 언제 반드시 쓰나

- 사용자가 UI를 보고 “뭔가 이상하다”, “수치 좀 물어봐”, “askUserQuestion 형식으로 정하자”, “피그마처럼 맞추자”라고 할 때
- spacing, sizing, grouping, alignment, chip/tag layout, top bar gap, status style 같은 **시각적 미세조정**이 필요한데 구두 설명만으로 오해가 생길 때
- 첫 구현 후 사용자가 “내 의도는 그게 아니다”라고 정정할 때
- “이름/주소는 한 묶음”, “태그는 따로”, “이건 badge가 아니라 dot+text”처럼 **구조적 grouping 오류**가 드러났을 때

이 스킬은 단순히 숫자만 묻는 도구가 아니다. **UI 온톨로지**를 묻는다. 사용자가 말한 요소가 “한 줄”, “한 묶음”, “상하 분리”, “좌우 끝 정렬” 중 무엇인지 먼저 확정한 뒤, 그 다음에 dp/sp를 묻는다.

---

## 역할 경계

- 기본 모드에서는 **질문만 한다**
- 구현은 사용자가 “고쳐”, “반영”, “수정”이라고 명시했을 때만 한다
- 사용자가 이미 구현 중인 세션에 있다면, 먼저 현재 이해를 2~4줄로 정리한 뒤 질문 라운드로 들어간다

---

## 질문 도구 규칙

### 반드시 지킬 것

- 모든 질문은 `askUserQuestion` 성격의 **구조화된 질문 도구**로 보낸다
- 한 번에 **2~6개** 질문만 묶는다
- 각 옵션은 짧고 클릭 가능해야 한다
- `description`에는 **왜 이 선택이 중요한지**와 **시각적 결과**를 적는다
- 의존 관계가 있으면 라운드를 분리한다

### 하지 말 것

- “어떻게 할까요?” 같은 open-ended 질문 남발
- 처음부터 모든 dp/sp를 한 번에 다 묻기
- 사용자가 이미 말한 것을 다시 묻기
- 1차 답변만 듣고 구조를 확정했다고 착각하기

---

## 인터뷰 흐름

## Round 0 — 현재 이해 요약

질문 전에 짧게 정리한다.

형식:

```markdown
지금 내가 이해한 건 이거다.

- 태그: 한 줄 + 가로 스크롤
- 칩: 텍스트 길이만큼
- 이번 범위: 카드 전체 수치 조정

이제 진짜 갈라지는 디테일부터 묻겠다.
```

목적:
- 사용자가 “맞아 / 아니야”를 빠르게 판단하게 한다
- 첫 오해를 초반에 줄인다

---

## Round 1 — 구조를 먼저 잠근다

수치보다 먼저 **구조**를 묻는다.

우선순위:

1. 줄바꿈 규칙
2. grouping 규칙
3. 좌우 정렬 규칙
4. 상태 표현 방식

예시:

```markdown
1. 태그는 어떻게 배치할까?
- 자동 줄바꿈
- 한 줄 + 가로 스크롤
- 최대 2줄

2. 이름/주소/태그 관계는?
- 이름/주소 한 묶음 + 태그 별도
- 이름/태그 한 묶음 + 주소 별도
- 세 줄 독립 배치

3. 상태 표시는?
- dot + plain text
- badge/pill
- 텍스트만
```

규칙:
- 사용자가 “묶음”, “덩어리”, “붙어있어야” 같은 표현을 쓰면 **grouping 질문을 최우선**으로 올린다
- 사용자가 “왜 태그가 됐냐”, “왜 pill이냐”라고 하면 **shared component 오용**을 의심하고 재질문한다

### Round 1.5 — block composition 라운드 (반드시 추가)

사용자가 구현 후 "뭔가 블록 조합이 다르다", "이건 한 덩어리여야 한다", "위아래 블록이 붙어야 한다"고 하면,
수치 질문으로 넘어가기 전에 **block composition 라운드**를 반드시 넣는다.

이 라운드의 목적은 화면을 **semantic block 단위**로 다시 자르는 것이다.
즉, 요소를 줄 단위가 아니라 아래처럼 블록 단위로 재질문한다.

- 상단 identity block (프로필 / 이름 / 별 / 날짜 / 액션)
- media block (사진)
- meta block (옵션 / 촬영지)
- content block (본문 / 더보기)
- reaction block (좋아요 / 보조 액션)

질문 원칙:

1. 먼저 **몇 블록 구조인지** 묻는다
2. 그 다음 **어떤 블록끼리 붙어야 하는지** 묻는다
3. 마지막으로 **블록 사이 간격 강도**를 묻는다

좋은 질문 예시:

```markdown
1. 카드 구조는 어느 쪽이 맞아?
- 상단 / 사진 / 메타 / 본문 / 반응 = 5블록
- 상단 / 사진 / 메타+본문 / 반응 = 4블록
- 상단+사진 / 메타 / 본문 / 반응 = 4블록

2. 메타와 본문 관계는?
- 하나의 정보 블록처럼 붙어야 함
- 분리된 두 블록이어야 함
- 붙어있되 시각적 위계만 다르게

3. 상단 identity block 안에서 날짜/삭제는?
- 이름/별과 별도 우측 블록
- 같은 상단 블록 안이지만 더 느슨하게
- 사실 아래 줄로 빠져야 함
```

핵심 규칙:

- 사용자가 `간격`이라고 말해도 실제 문제는 **블록 조합 오류**일 수 있으니 먼저 composition을 재확정한다
- 사용자가 `이름/별/삭제가 따로 논다`고 하면, spacing 질문이 아니라 **block cohesion 질문**으로 되돌린다
- 이후 수치 라운드에서는 반드시 `이 블록 내부 값`인지 `블록 간 값`인지 구분해서 묻는다

---

## Round 2 — 대표 수치만 잠근다

구조가 잠긴 뒤에만 수치를 묻는다.

질문 묶음 추천:

- 썸네일 크기 / radius
- 카드 좌우 패딩 / 상하 패딩
- 타이포 핵심 2개 (이름, 보조정보)
- 칩 높이 / 칩 패딩 / 칩 간격
- 첫 카드 top gap 여부

좋은 질문 예시:

```markdown
1. 썸네일 크기
- 80x80
- 88x88
- 96x96

2. 카드 좌우 여백
- 16dp
- 20dp
- 24dp

3. 칩 높이
- 28dp
- 30dp
- 32dp
```

원칙:
- 한 라운드에 대표 수치만 묻고, 나머지는 구현 후 미세조정으로 넘긴다
- 사용자가 애매하게 답하면, 그 답을 **구현값으로 번역해서 다시 확인**한다

### Round 2.5 — visual audit 라운드 (반드시 추가)

사용자가 "아직 하나도 확인 안 됨", "폰트/별/간격/색/radius 다 다시 보자"라고 하면,
대표 수치 라운드 다음에 **세부 visual audit 라운드**를 넣는다.

이 라운드의 목적은 구조가 아니라 **현재 구현의 실제 시각 요소 하나하나**를 확인하는 것이다.

우선순위:

1. 프로필 이미지 크기 / 테두리 / 이미지-텍스트 간격
2. 제목 폰트 크기 / 굵기 / 라인 높이
3. 제목과 별 사이 간격 / 별 크기 / 별 간격
4. 날짜/삭제 pill 정렬 관계
5. 삭제 pill의 색 / 높이 / radius / 내부 패딩
6. 사진 radius / 카드 radius / 카드 보더 강도
7. 메타(label-value) 폰트 크기 / 간격 / 줄간 거리
8. 본문 폰트 크기 / 더보기 텍스트 톤

질문 묶음 규칙:

- 한 번에 **같은 블록**만 묻는다. 예: `프로필 블록` 4문항, `삭제 pill` 4문항
- 각 질문은 가능하면 **현재값 vs 후보값** 형태로 묻는다
- 가능하면 질문 앞에 **현재 구현값을 짧게 요약**한다
- 사용자가 숫자를 자유 입력하면, 다음 응답에서 반드시 **구현값 번역**으로 재확인한다

예시:

```markdown
지금 프로필 블록의 현재값은 이렇다.

- 프로필 37dp
- 프로필-이름 간격 10dp
- 제목-별 간격 2dp
- 별 크기 11dp

이제 이 블록만 다시 잠그자.
```

좋은 질문 예시:

```markdown
1. 프로필 썸네일 크기
- 32dp
- 37dp (현재값)
- 40dp

2. 이름 폰트 크기감
- 지금보다 작게
- 현재값 유지
- 지금보다 크게

3. 이름과 별 사이 간격
- 2dp (현재값)
- 4dp
- 6dp

4. 별과 별 사이 간격
- 거의 붙게
- 현재값 유지
- 더 벌리기
```

삭제 pill 전용 라운드 예시:

```markdown
지금 삭제 pill의 현재값은 이렇다.

- 빨간 배경
- radius 5dp
- 좌우 6dp / 상하 3dp
- 캡션 폰트

이제 색/크기/모서리만 잠그자.
```

```markdown
1. 삭제 pill 색감
- 지금보다 진하게
- 현재값 유지
- 지금보다 덜 강하게

2. 삭제 pill 높이감
- 더 낮게
- 현재값 유지
- 더 높게

3. 삭제 pill radius
- 5dp (현재값)
- 8dp
- 완전 pill형

4. 삭제 pill과 날짜 간격
- 더 붙게
- 현재값 유지
- 더 벌어지게
```

핵심 원칙:

- **현재 구현값을 공개한 뒤 묻는다** — 사용자가 무엇을 바꾸는지 알아야 한다
- "폰트 이상함" 같은 추상 피드백은 반드시 `제목 / 메타 / 본문 / 액션` 중 어느 타이포 블록인지 분해한다
- `별`은 항상 `크기`, `별 사이 간격`, `제목과의 거리`를 따로 묻는다
- `radius`는 항상 `카드`, `사진`, `pill` 중 대상을 분리해 묻는다
- 같은 라운드에서 typography와 spacing을 섞더라도, **한 visual block 안에서만** 묻는다

예:

```markdown
좋아. 내가 구현값으로 번역하면 이렇다.

- 카드 상하 패딩: 20dp
- 칩 높이: 30dp
- 칩 간격: 4dp

이 해석이 맞으면 그대로 반영한다.
```

---

## Round 3 — 구현 후 어긋남 진단

사용자가 “아직 이상하다”고 하면, 같은 질문을 반복하지 않는다.
이 단계에서는 **실패 원인 유형**을 먼저 분류한다.

분류 축:

1. padding/rhythm 문제
2. width/space distribution 문제
3. grouping 문제
4. wrong shared component 문제
5. state/style mismatch 문제
6. typography/icon rhythm 문제
7. chip color/radius 문제

이때는 아래처럼 말한다.

```markdown
지금 문제는 수치보다 구조에 가깝다.

- 이름/주소가 한 묶음인지
- 태그가 별도 묶음인지
- 상태가 badge가 아니라 inline status인지

이 세 가지만 다시 잠그자.
```

즉, **2차 라운드는 재측정이 아니라 재해석 라운드**다.

---

## Round 4 — 추가 요구 패턴 처리

이번 세션에서 자주 나온 패턴은 아래와 같다.

### Pattern A — “처음엔 맞다고 했는데, 구현 보니 다름”

대응:
- 이전 답변을 고집하지 말고, **오해한 구조를 인정**한다
- “아 그러면 구조를 이렇게 다시 이해하면 되냐”로 리프레임한다

### Pattern B — “이름/주소는 한 묶음이고, 그 묶음과 칩이 위아래로 붙어야 함”

대응:
- 3단 구조로 묻지 말고 **2묶음 구조**로 다시 묻는다
- 정보행을 줄 단위가 아니라 **semantic block** 단위로 다룬다

### Pattern C — “왜 이게 badge가 됐냐”

대응:
- 공통 컴포넌트 재사용이 진짜 맞는지 다시 묻는다
- 사용자가 visual language를 명확히 말하면, 공통 컴포넌트보다 **요구된 시각 언어 우선**

### Pattern D — “우선순위가 중요한가?”

대응:
- 사용자가 한 번에 전체 범위를 맞추길 원하면 우선순위 질문을 접고, **대표 수치 패키지**로 전환한다

---

## 질문 설계 규칙

### 1. 수치 질문 전에 구조 질문

Bad:
- “칩 높이 몇 dp?”

Good:
- “칩이 이름/주소와 같은 묶음인가, 별도 묶음인가?”

### 2. 사용자의 말을 구현값으로 번역해서 재확인

Bad:
- “넉넉하게” → 혼자 해석 후 바로 구현

Good:
- “내가 구현값으로 번역하면 상하 패딩 20dp, 칩 높이 30dp다. 맞으면 반영.”

### 3. 실패 후에는 원인 분류 질문

Bad:
- “다시 어느 dp가 이상해?”

Good:
- “지금 문제는 padding인가, grouping인가, status style인가?”

### 5. visual audit는 블록 단위로 묻기

Bad:
- “프로필, 폰트, 별, pill 다 몇 dp?”

Good:
- “프로필 블록 먼저 잠그자: 프로필 크기 / 프로필-이름 간격 / 제목-별 간격 / 별 크기”

### 6. 현재 구현값을 기준선으로 제시하기

Bad:
- “radius 얼마가 맞아?”

Good:
- “현재 사진 radius는 5dp다. 5dp 유지 / 8dp / 더 각지게 중 뭐가 맞아?”

### 4. 질문은 항상 비교 가능하게

옵션은 이렇게 만든다.

- `한 줄 + 가로 스크롤`
- `자동 줄바꿈`
- `최대 2줄`

이처럼 **시각 결과가 머리에 그려지는 옵션**이어야 한다.

---

## 출력 형식

각 라운드 후에는 항상 짧게 합의 사항을 정리한다.

형식:

```markdown
좋아. 지금까지 잠긴 건 이거다.

- 태그: 한 줄 + 가로 스크롤
- 칩: 텍스트 길이만큼
- 구조: 이름/주소 한 묶음, 칩 별도 묶음

이제 남은 건 썸네일/패딩/칩 수치만 정하면 된다.
```

질문 라운드가 끝나면:

```markdown
확정값:
```

## 안티패턴

| 안티패턴 | 왜 문제인가 | 수정 |
|---|---|---|
| 숫자만 계속 묻기 | 구조 오해를 못 잡음 | grouping/정렬 먼저 묻기 |
| 첫 답변을 절대화 | 구현 후 정정 요구를 놓침 | 2차/3차 라운드 허용 |
| 잘못된 공통 컴포넌트 재사용 전제 | 사용자가 원하는 visual language 훼손 | shared component 적합성 재검토 |
| 너무 많은 질문 한 번에 던지기 | 사용자가 피로해짐 | 라운드별 2~6개만 |
| “뭐가 이상하죠?” 식 open 질문 | 답이 추상적이라 구현 불가 | 비교 가능한 옵션으로 재구성 |

## 빠른 템플릿

### 템플릿 1 — 초기 라운드

```markdown
이건 구조 먼저 잠그는 게 맞다.

1. 태그 배치
2. 요소 grouping
3. 상태 표현 방식

이 3개만 먼저 정하자.
```

### 템플릿 2 — 수치 라운드

```markdown
구조는 잠겼다. 이제 대표 수치만 묻겠다.

- 썸네일 크기
- 카드 패딩
- 칩 높이/간격
```

### 템플릿 2.5 — 세부 visual audit 라운드

```markdown
좋아. 구조는 맞는데, 아직 visual detail은 안 잠겼다.

지금 이 블록의 현재 구현값은 이렇다.

- 프로필 37dp
- 제목-별 간격 2dp
- 별 크기 11dp
- 삭제 pill radius 5dp

이제 이 블록만 다시 묻겠다.
```

### 템플릿 3 — 오진 교정 라운드

```markdown
좋아. 내가 구조를 잘못 이해했다.

- 이름/주소는 한 묶음
- 칩은 별도 묶음
- 상태는 badge가 아니라 inline status

이 기준으로 다시 묻겠다.
```

## 최종 원칙

UI askUserQuestion 인터뷰의 목적은 “숫자 수집”이 아니다.

> **시각적 의도를 구조 → 수치 → 교정 순서로 잠그는 것**

첫 라운드는 broad,
두 번째는 measurable,
세 번째는 correction,
마지막은 implementable state여야 한다.
