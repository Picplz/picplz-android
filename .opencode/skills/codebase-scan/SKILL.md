# codebase-scan — 코드베이스 스캔

새 코드를 기존 코드베이스에 맞게 작성하기 위해 **알아야 할 것**을 빠르게 파악한다.

## 사용법

```
/codebase-scan [경로]
```

> 경로 미지정 시 현재 디렉토리. `/spec-interview` 전에 실행하면 Context Clarity가 자동으로 올라감.

---

## 파이프라인 위치

```
codebase-scan → spec-interview → spec → test-spec → ralph
 ↑ 여기
```

출력물은 `spec-interview`의 **Context Clarity (15%)** 차원을 채우는 근거가 된다.

---

## 세 가지 질문

| # | 질문 | 목적 |
|---|------|------|
| 1 | **여기에 뭐가 있나?** | 스택, 구조, 의존성 파악 |
| 2 | **여기선 어떻게 쓰나?** | 패턴, 컨벤션 파악 |
| 3 | **여기서 뭘 하면 안 되나?** | 금지 패턴, 기존 유틸 중복 방지 |

---

## 실행 방법

### 1. 여기에 뭐가 있나? (What Exists)

설정 파일을 스캔하여 기술 스택과 구조를 파악한다.

**스캔 대상:**

```bash
# 스택 감지
package.json / build.gradle / Cargo.toml / pyproject.toml / go.mod

# 타입/빌드 설정
tsconfig.json / vite.config.* / webpack.config.* / next.config.*

# 프레임워크 단서
app/ or pages/ (Next.js) / src/app/ (FSD) / components/ (flat)
```

**추출 항목:**

| 항목 | 소스 | 예시 |
|------|------|------|
| 언어 + 버전 | tsconfig, package.json engines | TypeScript 5.4, Node 20 |
| 프레임워크 | dependencies | React 19, Next.js 15, Vite 6 |
| 아키텍처 패턴 | 폴더 구조 | FSD (app/pages/widgets/features/entities/shared) |
| 상태 관리 | dependencies + imports | Jotai, Zustand, Redux, TanStack Query |
| API 레이어 | shared/api/ 또는 lib/api/ | fetch wrapper, axios instance, tRPC |
| 스키마/검증 | dependencies | Zod, Yup, io-ts |
| DB/ORM | dependencies | Prisma, Drizzle, Supabase |
| 테스트 프레임워크 | devDependencies + config | Vitest, Jest, Playwright |

**출력:**

```yaml
stack:
  language: TypeScript 5.4
  runtime: Node 20
  framework: React 19 + Vite 6
  architecture: FSD (Feature-Sliced Design)
  state: Jotai (atoms) + TanStack Query (server state)
  api: fetch wrapper (shared/api/client.ts)
  schema: Zod
  test: Vitest
```

### 2. 여기선 어떻게 쓰나? (How It's Written)

기존 코드의 패턴과 컨벤션을 파악한다.

**스캔 대상:**

```bash
# 린트/포맷 설정
.eslintrc* / eslint.config.* / .prettierrc* / biome.json

# 기존 코드 샘플링 (최근 수정 파일 5-10개)
git log --diff-filter=M -10 --name-only

# 네이밍 패턴
ls src/**/*.ts | head -20
```

**추출 항목:**

| 항목 | 방법 | 예시 |
|------|------|------|
| 파일 네이밍 | 기존 파일명 패턴 | kebab-case (`user-profile.ts`) vs camelCase |
| 컴포넌트 패턴 | 샘플 컴포넌트 확인 | FC vs function, default vs named export |
| 훅 패턴 | hooks/ 디렉토리 확인 | `use*.ts` 네이밍, 커스텀 훅 구조 |
| 임포트 순서 | eslint import rules / 샘플 | external → internal → relative |
| 에러 핸들링 | try-catch 패턴, error boundary | Result type, throw, ErrorBoundary |
| 비동기 패턴 | API 호출 코드 확인 | async/await + TanStack Query, SWR |
| 타입 관리 | types/ 위치, 네이밍 | `*.types.ts`, `*.d.ts`, 인라인 |

**출력:**

```yaml
conventions:
  file_naming: kebab-case
  component: named export + function declaration
  hooks: use*.ts in hooks/ directory
  imports: external → @shared → @features → relative
  async: TanStack Query (useQuery/useMutation) + Zod parse
  types: co-located (*.types.ts next to component)
  error: Zod safeParse + QueryErrorResetBoundary
```

### 3. 여기서 뭘 하면 안 되나? (What to Avoid)

기존에 이미 있는 것, 금지된 패턴, lint 규칙을 파악한다.

**스캔 대상:**

```bash
# 금지 규칙
grep -r "no-restricted" eslint.config.* .eslintrc*
grep -r "ban" biome.json

# 기존 유틸리티 (중복 방지)
ls shared/lib/ shared/utils/ shared/helpers/ lib/ utils/ 2>/dev/null

# AGENTS.md / CONTRIBUTING.md 규칙
cat AGENTS.md CONTRIBUTING.md CONVENTIONS.md 2>/dev/null | head -100

# 타입 엄격도
grep "strict" tsconfig.json
```

**추출 항목:**

| 항목 | 방법 | 예시 |
|------|------|------|
| 금지된 API/패턴 | eslint no-restricted-* | `console.log` 금지, `any` 금지 |
| 이미 있는 유틸 | shared/lib/ 스캔 | `formatDate`, `cn`, `fetcher` — 새로 만들지 말 것 |
| 이미 있는 훅 | shared/hooks/ 스캔 | `useDebounce`, `useMediaQuery` |
| 이미 있는 컴포넌트 | shared/ui/ 스캔 | `Button`, `Modal`, `Table` |
| 문서화된 규칙 | AGENTS.md, CONTRIBUTING.md | "FSD 레이어 위반 금지", "직접 fetch 금지" |
| 타입 엄격도 | tsconfig strict 옵션 | strict: true, noUncheckedIndexedAccess 등 |

**출력:**

```yaml
avoid:
  banned_patterns:
    - "console.log (use logger)"
    - "any type (strict mode)"
    - "direct fetch (use shared/api/client)"
  existing_utils:
    - shared/lib/format-date.ts
    - shared/lib/cn.ts (className merge)
    - shared/hooks/use-debounce.ts
  existing_ui:
    - shared/ui/Button, Modal, Table, Input
  documented_rules:
    - "FSD: pages → features → entities → shared (역방향 import 금지)"
    - "API: shared/api/client.ts 통해서만 호출"
  type_strictness: strict + noUncheckedIndexedAccess
```

## 최종 출력: Codebase Context

세 섹션을 합쳐 하나의 컨텍스트 문서로 출력:

```yaml
# === Codebase Context ===
# Generated by /codebase-scan at {timestamp}

stack:
  language: TypeScript 5.4
  runtime: Node 20
  framework: React 19 + Vite 6
  architecture: FSD
  state: Jotai + TanStack Query
  api: shared/api/client.ts (fetch wrapper)
  schema: Zod
  test: Vitest

conventions:
  file_naming: kebab-case
  component: named export + function declaration
  hooks: use*.ts co-located
  imports: external → @shared → @features → relative
  async: TanStack Query + Zod safeParse
  types: co-located *.types.ts

avoid:
  banned: [console.log, any, direct fetch]
  existing_utils: [format-date, cn, use-debounce]
  existing_ui: [Button, Modal, Table, Input]
  rules: ["FSD 역방향 import 금지", "API는 client.ts 경유"]
  type_strictness: strict
```

## spec-interview 연동

이 출력은 `/spec-interview`의 **Context Clarity 차원**에 직접 기여한다:

- 스캔 결과가 있으면 → Context Clarity 기본 0.7+ 시작 (이미 많이 알고 있음)
- 스캔 없이 인터뷰 시작하면 → Context Clarity 0.3~0.5에서 시작 (질문으로 채워야 함)

또한 `spec-interview`가 **확인형 질문**을 만들 때 이 컨텍스트를 활용:
- ✅ "Jotai atom이 있네요. 새 상태도 atom으로 관리할까요?"
- ❌ "상태 관리 뭐 쓰세요?"
